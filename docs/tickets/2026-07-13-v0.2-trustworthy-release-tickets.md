## T00: Prove Claude Hook Replacement Viability

**Goal:** Prove that the current supported Claude Code version accepts
`PostToolUse.hookSpecificOutput.updatedToolOutput` for a built-in command tool
and that the model-visible tool result contains the governed receipt rather than
the original middle sentinel.

**Scope:** Official Claude Hook reference evidence, current npm-packed artifact,
a temporary project, temporary npm prefix, temporary ledger, Claude Code CLI
`2.1.206` or the current installed version, a minimal built-in-tool output-shape
adapter fix in `claude_hook.py` with its regression test, and a reproducible
evidence report under `docs/evidence/`.

**Allowed Files:** `src/token_governance/claude_hook.py`,
`tests/test_claude_hook.py`, and T00 evidence. No governance semantics change.

**Out of Scope:** Governance redesign, paid A/B benchmark, package publishing,
or permanent Claude configuration changes.

**Owner Layer:** Claude Hook integration adapter and viability evidence.

**Dependencies:** Approved design and the user's authorization for at most two
minimal real Claude calls.

**Implementation Notes:** Build the current tarball, install it under a temporary
npm prefix, and create a temporary Claude project. The generator must create a
random sentinel at runtime so the sentinel is absent from the prompt, command,
and source file. Capture Claude CLI version, exact command, hook response,
model-visible answer, relevant redacted session/debug evidence, and cleanup
result. Identify the documented/observed structured Claude session or stream
field that represents the tool result presented to the model, then assert that
field directly. A natural-language model answer is supporting evidence only and
cannot satisfy the gate. Never include authentication data or unrelated
transcript content.

Root-cause evidence from the first real call: the Hook returned a string for
`updatedToolOutput`; current official docs require Bash replacement output to
match `{stdout, stderr, interrupted, isImage}`. Claude therefore ignored the
replacement and used the original. Apply only the shape correction under TDD
before the second and final authorized model call.

**Forbidden Actions:** Do not modify global Claude settings, expose credentials,
make more than two model calls, publish packages, commit temporary transcripts,
or interpret a local JSON unit test as viability proof.

**Acceptance Criteria:** Official schema supports the field; the real Claude
session's machine-read model-visible tool-result field contains the receipt
marker and excludes the random middle sentinel; the original remains
retrievable through the temporary ledger; all temporary processes and files are
cleaned.

**Verification:** `npm pack`; temporary-prefix `npm install -g`; `claude
--version`; one minimal `claude -p` run with structured/debug output; direct
ledger retrieval; `git status --short` shows only planned repository artifacts.

**Stop Conditions:** Stop the entire implementation if Claude is unavailable
after two allowed calls, if the Hook response is ignored, if original output
still enters model context, or if official schema evidence conflicts with the
design.

## T01: Add Typed Configuration and Governance Contracts

**Goal:** Establish the singular typed source of truth for runtime configuration
and immutable governance request/decision/result contracts.

**Scope:** New configuration/contracts modules, `token-governance.config.example.json`,
CLI config loading helpers, and focused tests.

**Allowed Files:** New `src/token_governance/config.py`,
`src/token_governance/contracts.py`, `tests/test_config.py`,
`tests/test_contracts.py`, and `token-governance.config.example.json`.

**Out of Scope:** Shaping logic, ledger migration, installer writes, MCP process
lifecycle, or documentation rewrite.

**Owner Layer:** Typed `GovernanceConfig` and governance contract types.

**Dependencies:** T00.

**Implementation Notes:** Use closed enums and frozen dataclasses. Config is
strict, resolves relative paths from the config file directory, supports only
built-in strategies, bounds numeric values, and uses precedence CLI override,
explicit config, defaults. `tool_input` is memory-only by contract.
Define the shaper matcher/registry protocol and ledger/event port here so later
tickets inject implementations without changing policy ownership.

**Forbidden Actions:** No free-form strategy/content/risk values, arbitrary
user regex execution, compatibility alias for generic head/tail compression, or
adapter-specific config parsing.

**Acceptance Criteria:** Good config loads identically for all adapters; unknown
fields and invalid bounds fail with fixed messages; path resolution is cwd
independent; contract types reject invalid enums and mutable data.

**Verification:** Add tests first and observe RED; run
`python -m pytest -q tests/test_config.py tests/test_contracts.py`; run
`python -m pytest -q` after GREEN.

**Stop Conditions:** Stop if official Hook fields cannot map to the contract
without loss or if current public config semantics require an unapproved
compatibility path.

## T02: Add Privacy-Safe Secret Detection and Fail-Open Classification

**Goal:** Ensure secret-like, unknown, oversized, or failed classifications pass
through without raw persistence while producing only payload-free event metadata.

**Scope:** Secret detector, policy selection, reason/risk enums, engine decision
path, bounded payload/deadline behavior, and safety tests.

**Allowed Files:** New `src/token_governance/secret_detector.py`, existing
`src/token_governance/policy.py`, `src/token_governance/core.py`, and new
`tests/test_secret_detector.py`, `tests/test_policy_v2.py`.

**Out of Scope:** Actual content transformation, receipt storage, Gateway, or
docs beyond developer-facing type notes.

**Owner Layer:** `GovernanceEngine` orchestrating `SecretDetector` and
`PolicyEngine`.

**Dependencies:** T01.

**Implementation Notes:** Fixed bounded patterns cover common API keys,
authorization headers, private keys, passwords, and environment names. Matched
values never enter messages. Adapters provide raw fields only; engine derives
command family. Unknown/ambiguous/malformed/oversized/deadline cases fail open.

**Forbidden Actions:** Do not persist or hash secret payloads, log free-form
tool input, expose matched values, accept arbitrary regex, or mark unavailable
classification as low risk.

**Acceptance Criteria:** Secret corpus covers first/middle/end, Unicode and
stdout/stderr/tool-input cases; a spy ledger proves receipt insertion is never
called; metadata events contain closed values only. Policy selection is tested
against a fake injected matcher registry and never imports concrete shapers.

**Verification:** TDD RED-GREEN with
`python -m pytest -q tests/test_secret_detector.py tests/test_policy_v2.py`, then
full pytest. Include injected detector error, timeout, malformed payload, and
size-limit cases.

**Stop Conditions:** Stop if any secret reaches a persistent/free-form surface
or if fail-open behavior depends on host-specific undefined behavior.

## T03: Migrate Ledger to Schema v2

**Goal:** Provide transactional v0.1-to-v0.2 migration, transformed-only
receipts, payload-free governance events, delivery state, retention, integrity,
and bounded concurrency.

**Scope:** `ContextLedger`, a committed v0.1 fixture containing synthetic legacy
rows, migration/prune/purge APIs, tests, and CLI-facing result shapes only.

**Allowed Files:** `src/token_governance/ledger.py`, new
`tests/test_ledger_v2.py`, and a synthetic fixture under `tests/fixtures/`.

**Out of Scope:** Hook formatting, shapers, installer, cloud storage, encryption,
or secure erasure.

**Owner Layer:** `ContextLedger`.

**Dependencies:** T01.

**Implementation Notes:** Use `PRAGMA user_version=2`, one migration transaction,
nullable v2 fields for legacy rows, event table without raw/hash/free text,
prepared/emitted delivery state, bounded busy timeout, UTC cutoff, explicit
prune, and explicit confirmed legacy purge. Verify hashes during retrieval.

**Forbidden Actions:** No automatic legacy deletion, implicit Hook-path prune,
unbounded retries/sleeps, migration without rollback test, or claims of secure
SQLite erasure.

**Acceptance Criteria:** Real v0.1 fixture migrates and remains retrievable;
failed migration rolls back byte-for-byte logical state; prune removes expired
receipts and linked/unlinked events; purge requires confirmation; concurrent
writes are bounded; stats separate prepared and emitted candidate savings.

**Verification:** TDD with `python -m pytest -q tests/test_ledger_v2.py`,
concurrent process/thread tests, full pytest, and SQLite queries checking
schema/version/indexes.

**Stop Conditions:** Stop if migration would destroy existing receipts or if
Windows file/locking behavior cannot be bounded.

## T04: Implement Safe Shapers and Independent Preservation Verification

**Goal:** Replace generic sampling with registered, applicability-checked,
reconstructable transformations for repetitive logs, test output, and build
output.

**Scope:** Shaper registry, applicability matchers, collapsed-span model,
independent verifier, removal of generic head/tail summarizer, and fixtures.

**Allowed Files:** New `src/token_governance/shapers.py`,
`src/token_governance/preservation.py`, `tests/test_shapers.py`, and
`tests/test_preservation.py`. Do not modify `PolicyEngine` or adapter modules.

**Out of Scope:** Hook/MCP/CLI wiring, ledger schema, unregistered content types,
source/diff/search summarization, or model-based summarization.

**Owner Layer:** Registered shapers and independent `PreservationVerifier`.

**Dependencies:** T02.

**Implementation Notes:** Explicit strategy remains subject to matcher. Exact
contiguous duplicates and registered progress units are the only deletable
units. Every other unique unit remains byte-for-byte. Verifier accounts for the
entire original sequence independently of shaper claims.

**Forbidden Actions:** No generic first/last sampling, semantic rewriting,
unique-line deletion, keyword-only proof, or risk assignment inside shapers.

**Acceptance Criteria:** Middle unique fact survives; source/JSON/unknown passes
through; duplicate logs shrink with exact count/order; pytest/build errors,
warnings, assertions, stderr and exit status remain exact; verifier rejects a
malicious/buggy shaper result.

**Verification:** Write and observe failing tests before implementation; run
`python -m pytest -q tests/test_shapers.py tests/test_preservation.py` and full
pytest after each GREEN cycle.

**Stop Conditions:** Stop if a proposed compression cannot be reconstructed or
verified without trusting the shaper.

## T05: Integrate the V2 Engine Across Hook, CLI, and Manual MCP Governance

**Goal:** Route Hook, CLI `govern`, and MCP `govern_context` through the same v2
engine and fail-open/persistence truth.

**Scope:** `core.py`, `claude_hook.py`, CLI governance command, MCP manual tool,
updated schemas, prepared/emitted receipt flow, and integration tests.

**Allowed Files:** `src/token_governance/core.py`, `claude_hook.py`, `cli.py`,
`mcp_server.py`, existing adapter tests, and new `tests/test_governance_v2.py`.

**Out of Scope:** MCP lifecycle changes, installer transaction, Gateway, or docs
rewrite.

**Owner Layer:** `GovernanceEngine`; adapters only normalize protocols.

**Dependencies:** T02, T03, T04.

**Implementation Notes:** Hook preserves stdout/stderr/exit status/tool input,
prebuilds response, commits receipt, emits response, then marks emitted. CLI/MCP
accept only registered strategy enum. Old `content_type` returns migration
guidance. Pass/secret/error cases create no receipt. T05 owns the read-only
install-state contract and Hook gate: only a manually prepared `complete` state
permits transformation; missing/incomplete state passes through. T07 later owns
creation, repair, and deletion of that state. CLI exact full export is also
owned here; T06 only changes MCP pagination.

**Forbidden Actions:** No adapter risk/classification logic, silent old-argument
compatibility, partial JSON output, or actual-savings claim from candidate stats.

**Acceptance Criteria:** All three adapters produce equivalent decisions for
the same request; Hook failures return valid fail-open behavior; emitted stats
exclude prepared rows; original is retrievable for transformed rows only.
Complete/incomplete install-state fixtures gate Hook behavior. The final secret
corpus is rerun through the real schema-v2 ledger and scans DB, journal, WAL,
events, Hook stdout and Hook stderr for matched values.

**Verification:** TDD with
`python -m pytest -q tests/test_governance_v2.py tests/test_claude_hook.py tests/test_cli.py tests/test_mcp_server.py`,
subprocess Hook smoke, CLI/MCP roundtrips, injected serialization/write/ledger
failures, privacy file scan, and full pytest.

**Stop Conditions:** Stop if Hook cannot fail open for an injected runtime error
or if adapter outputs diverge from engine truth.

## T06: Harden Standalone Restore MCP

**Goal:** Make the standalone MCP server lifecycle-correct and bound original
retrieval inside model context.

**Scope:** `mcp_server.py`, protocol version negotiation, notification dispatch,
paginated `retrieve_original`, integrity errors, tool schemas, and tests.

**Allowed Files:** `src/token_governance/mcp_server.py` and
`tests/test_mcp_server.py`. CLI exact export remains owned by T05.

**Out of Scope:** Gateway backend transport, resources/prompts, official SDK
rewrite, or encryption.

**Owner Layer:** Standalone `McpServer` protocol adapter; ledger owns data truth.

**Dependencies:** T03, T05.

**Implementation Notes:** Notifications never receive responses; initialized and
cancellation affect state. Requests before initialize fail. MCP retrieval uses
bounded offset/max_chars and truncation metadata. Old unbounded response
assumption gets migration error/guidance.

**Forbidden Actions:** No hard-coded unrelated protocol response, unbounded MCP
raw retrieval, raw exception leakage, or conversion of MCP tool errors into fake
success.

**Acceptance Criteria:** Initialize/initialized/tools list/call lifecycle works;
notifications are silent; incompatible version and illegal order fail
structurally; pagination and hash mismatch are correct; CLI full export remains
exact.

**Verification:** `python -m pytest -q tests/test_mcp_server.py`, strict protocol
subprocess tests, pagination/integrity tests, full pytest, and packed-artifact
MCP smoke later in T11.

**Stop Conditions:** Stop if the official protocol requires an unsupported
transport/lifecycle redesign beyond the approved tools-only boundary.

## T07: Make Claude Installation Persistent and Reversible

**Goal:** Provide stable global-install configuration, fail-open partial-install
state, read-only diagnosis, explicit repair, and safe uninstall.

**Scope:** CLI init/install/repair/uninstall/doctor, config/MCP/settings writes,
install-state file, ownership markers, global-vs-npx detection, failure injection,
and tests.

**Allowed Files:** New `src/token_governance/installer.py`, installer portions
of `src/token_governance/cli.py`, `tests/test_installer.py`, and installer cases
in `tests/test_cli.py`. Do not change Hook install-state read semantics from T05.

**Out of Scope:** npm publication, project-local runtime copying, ledger purge by
default, or unrelated Claude settings.

**Owner Layer:** Claude project installer.

**Dependencies:** T05, T06.

**Implementation Notes:** Order is config, MCP, Hook, complete state; uninstall
is Hook, MCP, state. Hook transforms only with complete state. Resolve stable
absolute global bin paths. Reject npx persistent installation without modifying
the project. `doctor` is read-only; repair/removal are explicit.

**Forbidden Actions:** No broad substring deletion, cross-file atomicity claim,
overwrite of user config, npx temp-path persistence, implicit purge, or global
Claude config mutation.

**Acceptance Criteria:** Idempotent install preserves unrelated entries;
failure at every write point leaves fail-open/recoverable state; doctor changes
no bytes; repair/uninstall touch exact owned entries; npx install is rejected;
paths with spaces/non-ASCII work.

**Verification:** `python -m pytest -q tests/test_installer.py tests/test_cli.py`,
TDD failure-injection tests, byte-for-byte config assertions, subprocess CLI
tests, full pytest, and packed global install smoke in T11.

**Stop Conditions:** Stop if installed bin location cannot be proven stable in a
new shell/process or if ownership cannot be distinguished safely.

## T08: Harden the Experimental MCP Gateway

**Goal:** Make the tools-only Gateway lifecycle bounded and honest while keeping
it outside the primary product promise.

**Scope:** Gateway client/backend state machines, independent initialize
negotiation, response routing, notifications, cancellation, cache invalidation,
timeouts, stderr ring, partial availability, shutdown, and tests/docs evidence.

**Allowed Files:** `src/token_governance/mcp_gateway.py`,
`tests/test_mcp_gateway.py`, strict backend fixtures under `tests/fixtures/`, and
machine-readable interoperability evidence under `docs/evidence/`. If the safe
fallback is required, T08 may remove only the Gateway bin entry from
`package.json`; T10 owns public `docs/gateway.md` and all other metadata.

**Out of Scope:** Resources/prompts proxying, official SDK packaging rewrite,
or broad compatibility claims.

**Owner Layer:** Gateway backend transport and Gateway protocol adapter.

**Dependencies:** T03, T06.

**Implementation Notes:** Gateway uses its own client identity/capabilities.
Transport errors remain JSON-RPC errors; backend `isError` tool results remain
tool results. Strict fixtures cover interleaving and failure. Record pinned,
credential-free reference servers when practical.

**Forbidden Actions:** No request before backend ACTIVE, stderr on protocol
stdout, unbounded blocking, response-by-next-line assumption, fake healthy
backend, or promotion out of experimental based only on fake fixtures.

**Acceptance Criteria:** All state transitions and illegal order cases are
covered; notifications do not satisfy requests; timeout/cancel/list-change/
partial failure/kill work; strict fixture passes. Real-backend gaps are recorded
and Gateway remains experimental. A backend fixture emits a synthetic secret on
stderr; the ring buffer, client-visible unavailable error, stdout, ledger and
evidence artifacts contain no matched value.

Two terminal states are valid: (A) all strict lifecycle/fault fixtures pass and
the experimental bin remains packaged; or (B) if strict safety cannot be
implemented within the approved dependency-free transport, the public Gateway
bin is removed from package distribution and the limitation/removal evidence is
recorded for T10. Only optional real reference-backend interoperability gaps may
remain while state A ships.

**Verification:** `python -m pytest -q tests/test_mcp_gateway.py`, focused TDD
Gateway suite, stderr secret/flood/timeout processes, privacy scan, full pytest,
and optional pinned reference-server smoke without credentials.

**Stop Conditions:** If hardening requires an official SDK and packaging
redesign, use terminal state B; do not leave a known-unsafe public bin while
claiming the ticket complete.

## T11: Add Version and Release Verification Tooling

**Goal:** Make one reproducible v0.2.0 artifact and prove clean installation and
release readiness before any irreversible publication.

**Scope:** `VERSION`, structured version/update/check scripts, release preflight,
tarball manifest/hash evidence, clean temporary global install, npx one-shot/
reject test, supported matrix commands, feature branch push, CI wait, and release
notes draft.

**Allowed Files:** `VERSION`, `scripts/release_*.py`, version fields/constants,
and release notes draft. Runtime changes are forbidden; failures return to the
owning prior ticket. Final tarball and preflight evidence live under ignored
`.tgl/release/v0.2.0/` so they cannot change the frozen commit.

**Out of Scope:** Actual tag, GitHub Release publication, npm publish, or changing
package contents after hashing.

**Owner Layer:** Release tooling.

**Dependencies:** T10.

**Implementation Notes:** Restore TLS verification before network auth checks.
Verify npm ping/whoami/owner authorization and GitHub push/tag/Release access
before tag. Version script updates package/pyproject/runtime constants from
`VERSION` and verifies equality. Controller commits all final files, pushes the
feature branch, and waits for CI. Only after that exact SHA is clean and verified
does it build/hash the tarball into `.tgl/release/v0.2.0/`. No tracked file may
change after the frozen SHA.

**Forbidden Actions:** No tag, publish, credential logging, disabled TLS,
`git add .`, dirty-worktree release, or artifact rebuild after final hash.

**Acceptance Criteria:** All version fields are 0.2.0; tarball clean install in a
new process passes init/install/doctor/hook/MCP/uninstall/reinstall; npx behavior
matches design; packed-artifact secret corpus/privacy scan passes; preflight
validates permissions; evidence records SHA/manifest/hash without secrets. The
feature branch is pushed and required CI completes successfully on that exact
SHA before the artifact is declared frozen.

**Verification:** Canonical release-check commands, packed artifact matrix,
privacy scan, full pytest, npm pack, git diff/status, feature branch push, and
required CI completion on the frozen SHA.

Canonical commands and evidence contracts:

- `python scripts/release_set_version.py 0.2.0`
- `python scripts/release_check.py --expected-version 0.2.0 --phase preflight --output .tgl/release/v0.2.0/preflight.json`
- `python scripts/release_verify_package.py --tarball .tgl/release/v0.2.0/token-governance-layer-0.2.0.tgz --output .tgl/release/v0.2.0/package.json`
- all scripts return 0 only on success and emit schema version `1`, frozen Git
  SHA, version, tarball SHA-256 where applicable, checks, and no credentials.

**Stop Conditions:** Stop before tag on TLS warning, npm/GitHub auth failure,
dirty worktree, CI failure, version/hash drift, or unexpected private metadata.

## T12: Publish v0.2.0

**Goal:** Publish the exact verified commit and tarball to GitHub and npm, then
prove all public references agree.

**Scope:** Fast-forward main integration of the already CI-verified frozen SHA,
`v0.2.0` tag, draft then published GitHub Release, npm publish of the recorded
tarball, GitHub topics, and post-release verification.

**Out of Scope:** Any code/doc modification after artifact hash, force pushes,
history rewriting, unrelated repository changes, or a second v0.2.0 artifact.

**Owner Layer:** Release controller.

**Dependencies:** T11 and all release preflight gates.

**Implementation Notes:** Fast-forward main to the same frozen SHA, tag it,
create draft Release,
publish exact tarball, verify npm, then publish Release. Same-hash npm retry is
allowed; changed contents require a new version/tag. If main cannot fast-forward,
stop and return to T11 on the new HEAD for complete revalidation and artifact
refreezing.

T12 consumes `.tgl/release/v0.2.0/preflight.json` and `package.json`, then runs:

- `python scripts/release_check.py --expected-version 0.2.0 --phase publish --output .tgl/release/v0.2.0/publish.json`
- `python scripts/release_verify_remote.py --expected-version 0.2.0 --expected-sha <frozen-sha> --tarball-sha256 <recorded-hash> --output .tgl/release/v0.2.0/remote.json`

No T12 step stages or commits files.

**Forbidden Actions:** No disabled TLS, leaked token, force push, tag movement,
publishing from dirty state, package rebuild, or deleting user changes.

**Acceptance Criteria:** `main`, tag, Release, npm `gitHead`, package version and
tarball hash agree; clean registry install works; GitHub topics/metadata visible;
local/remote status is clean and synchronized.

**Verification:** Git/GitHub API evidence, npm view/dist-tag/integrity evidence,
clean install from registry, final pytest/package smoke where applicable, and
final privacy scan.

**Stop Conditions:** On npm failure keep GitHub Release draft titled
`INCOMPLETE v0.2.0`; retry only identical hash. Any content change requires
v0.2.1. Stop on any auth, CI, integrity, or privacy failure.

## Sequencing Notes

T00 is the hard viability gate. T01 establishes contracts before any runtime
rewrite. T02 and T03 have separate owners and may be developed independently,
but controller reconciliation must complete both before T05. T04 depends on the
policy contract and can proceed while T03 is isolated. T05 serially integrates
all governance truth. T06 and T08 touch MCP protocol code and remain serial in a
shared worktree. T07 follows the stable Hook/MCP contracts. T09 can run after
T05 but must be reconciled against final T07/T08 behavior before T10. T10 and
T11 are serial because public docs and release artifacts must reflect final
runtime truth. T12 is exclusively controller-owned and begins only after all
verification and external permission gates pass.
