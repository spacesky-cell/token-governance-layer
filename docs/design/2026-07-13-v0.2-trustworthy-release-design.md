# 2026-07-13 v0.2 Trustworthy Release Design

## Goal

Deliver `token-governance-layer` v0.2.0 as a trustworthy, installable Claude Code
tool-output governance product. The release must default to fail-open behavior,
only transform output when a content-specific strategy can preserve required
facts, keep transformed originals locally for restoration, avoid storing
secret-like or passthrough payloads, and report evidence-backed risk and savings.
The GitHub and npm release must describe only behavior that is implemented and
verified.

The user-visible product boundary for v0.2.0 is:

> Automatic, reversible tool-output governance for Claude Code, with local
> receipts and conservative defaults.

The MCP Gateway remains an experimental tools-only integration until its
protocol lifecycle and real-backend interoperability checks pass.

## Non-Goals

- Do not claim generic support for every coding agent.
- Do not automatically summarize source-code reads, diffs, arbitrary search
  results, user instructions, security output, or unknown output shapes.
- Do not add cloud sync, a hosted service, a dashboard, team billing, or remote
  telemetry.
- Do not add encryption in v0.2.0. Secret-like payloads must instead bypass
  transformation and persistence. The plaintext local-storage boundary must be
  documented explicitly.
- Do not preserve the current generic head/tail summarizer as the automatic
  compatibility path.
- Do not rename the npm package or GitHub repository in this release.
- Do not publish from an npm identity that violates the user's privacy
  requirement.
- Do not create or copy a project-local runtime for npx. Persistent Claude
  integration requires a stable global npm installation in v0.2.

## Source of Truth and Owner Layer

### Governance semantics

`GovernanceEngine` owns the governance result: action, content, risk, receipt,
token counts, and notes. `PolicyEngine` owns whether a payload is eligible for a
specific shaping strategy. Content-specific shapers own only transformation and
fact-preservation validation.

Claude hooks, the CLI, MCP server, and MCP Gateway are adapters. They may map
protocol fields to a governance request, but they must not independently invent
risk, privacy, or persistence truth.

The executable contract is:

| Stage | Owner | Input | Output | Invariant |
| --- | --- | --- | --- | --- |
| Protocol normalization | Adapter | Host/MCP/CLI payload | `GovernanceRequest` | Preserve stdout, stderr, exit status, tool name, and tool input separately; never classify risk |
| Secret detection | `GovernanceEngine` through `SecretDetector` | Normalized request | Fixed reason codes only | Never return or log matched values |
| Strategy selection | `PolicyEngine` | Request metadata plus config | `PolicyDecision` | Only registered strategies; unknown or ambiguous means passthrough |
| Transformation | Registered shaper | Structured command result | `ShaperResult` | Shaper cannot set final risk or persistence |
| Preservation verification | `PreservationVerifier` called by engine | Original plus shaper result | `VerificationResult` | Independent exact-line/block checks; does not trust shaper claims |
| Persistence | `ContextLedger` | Verified result | Committed receipt ID | Commit succeeds before transformed content is returned |
| Final result | `GovernanceEngine` | All prior evidence | `GovernanceResult` | Only engine sets action, risk, receipt, and persistence outcome |

Required immutable data contracts:

```text
CommandResult(stdout, stderr, exit_code, interrupted)
GovernanceRequest(source_kind, tool_name, tool_input, command_result,
                  raw_text, payload_bytes, mode)
PolicyDecision(action, strategy, reason_code, confidence)
ShaperResult(content, candidate_protected_facts, diagnostics)
VerificationResult(ok, protected_fact_count, missing_fact_count, reason_code)
GovernanceResult(action, content, risk, reason_code, strategy, confidence,
                 preservation_check, token_before, token_after, tokens_saved,
                 receipt_id)
```

Enums are closed and validated. Free-form `content_type`, strategy names, risk,
and reason codes are not accepted from adapters or config.

`tool_input` is structured, remains in process memory only, and is inspected by
the engine for secret detection and derivation of a closed-enum
`command_family`. It is never written to a receipt, event, log, exception, or
debug artifact. Adapters do not derive `command_family`. CLI input has no command
metadata by default; `auto` may then select only the content-provable
`repetitive_log` strategy, while test/build strategies pass through unless the
CLI provides a validated registered command family.

### Configuration

A typed `GovernanceConfig` loaded from `token-governance.config.json` is the
single source of truth for thresholds, enabled strategies, ledger path,
retention, and protected-content behavior. The example file and installer must
use the same parser as runtime.

### Persistence

`ContextLedger` owns receipt persistence, retrieval, pruning, and aggregate
statistics. It must not receive secret-like or passthrough payloads under the
default policy.

### MCP transport

`McpServer` owns the standalone restore/audit MCP surface. `McpGateway` owns the
experimental tools-only proxy surface. Backend process lifecycle belongs to a
bounded backend transport class, not gateway business logic.

### Release truth

`package.json` and `pyproject.toml` must share version `0.2.0`. Git tag
`v0.2.0`, GitHub Release, npm version, README compatibility table, and package
metadata must be created from the same verified commit.

## Proposed Design

### 0. Integration viability gate

Before changing governance behavior, verify the primary Claude integration:

1. Pin and record the Claude Code version under test.
2. Confirm the official `PostToolUse` input/output schema and failure behavior.
3. Install the current packed npm artifact into a temporary prefix.
4. Run a real Claude session in a temporary project where a generator emits a
   random middle sentinel not present in the command or prompt.
5. Prove from the model-visible result and session/debug evidence that
   `updatedToolOutput` replaced the original result and that the receipt marker
   is visible.

This probe requires an available Claude model account and explicit approval for
the small external model cost. If it proves that built-in output cannot be
replaced, implementation stops and the product integration/positioning returns
for redesign. Local JSON shape tests are not sufficient evidence.

### 1. Conservative governance pipeline

The automatic pipeline is:

1. Claude hook extracts `tool_name`, structured `tool_input`, stdout, stderr,
   exit status, interruption state, and complete tool output without selecting
   a content type.
2. The adapter creates a structured governance request.
3. Secret-like detection runs before shaping or persistence.
4. `PolicyEngine` selects one explicit strategy or `passthrough`.
5. The selected shaper returns governed content plus preservation evidence.
6. The engine verifies that the result is smaller and that all protected facts
   are preserved.
7. Only a verified transformation creates a receipt and stores the original.
8. Any parser error, uncertainty, timeout, secret detection, failed
   preservation check, storage failure, or serialization failure returns the
   original output unchanged without a transformed response.

Automatic default strategies:

- `repetitive_log`: collapse repeated non-error lines and preserve exact error,
  warning, failure, traceback, path, assertion, and exit-status lines.
- `test_output`: preserve the final test summary and complete failure/error
  blocks; collapse repeated successful/noise lines only.
- `build_output`: preserve warnings, errors, failed targets, exit status, and
  final summary; collapse repeated progress output only.

Recognition is deterministic and ordered: `test_output`, `build_output`, then
`repetitive_log`. Each matcher requires registered command families and output
signatures. Mixed stdout/stderr is normalized into labeled sections and a
separate exit-status record before matching. Unknown commands, conflicting
matchers, missing exit status where required, and unrecognized mixed output pass
through. `PreservationVerifier` independently extracts exact error/warning/
failure/assertion/traceback/exit-status blocks from the original and requires
byte-for-byte presence in the candidate result.

An explicit strategy is only a candidate upper bound and never bypasses its
applicability matcher. `repetitive_log` may collapse only contiguous,
byte-identical lines and must preserve position, exact text, and occurrence
count. Test/build strategies define removable units as registered progress
tokens or contiguous exact duplicates; every other unique line remains exact.
`ShaperResult` includes a closed list of collapsed spans. The independent
verifier checks that every original unit is either present byte-for-byte or
covered by one allowed span with matching exact content and count. This prevents
the shaper from self-certifying deletion of a unique non-error business fact.

Default passthrough surfaces:

- `Read`, `Grep`, `Glob`, `LS`, `Task`, `WebFetch`, and `WebSearch`.
- Source code, diffs, arbitrary JSON, user instructions, security output, and
  unknown command output.

The automatic matcher is narrowed to `Bash|PowerShell`. Users may opt into
additional strategies in config after explicit documentation.

Resource bounds:

- `max_payload_bytes`: 2 MiB by default; larger payloads pass through and are
  not persisted.
- `max_stored_original_bytes`: 1 MiB by default; a larger original cannot be
  transformed because it would not be safely restorable.
- `hook_deadline_ms`: 2,000 by default; deadline exhaustion passes through.
- Custom strategies are not loadable. Config may enable or disable only
  built-in registered strategies.
- Secret patterns are fixed bounded matchers plus a bounded list of literal
  user markers. User-provided arbitrary regular expressions are not supported
  in v0.2.

Hook failure matrix:

| Failure | Hook stdout | Hook stderr | Persistence | Outcome |
| --- | --- | --- | --- | --- |
| Malformed/empty host payload | Valid `continue=true` when protocol allows; otherwise no output and exit 0 | Redacted reason code only | None | Host keeps original |
| Bad/missing config | `continue=true` | Redacted reason code | None | Original passes through |
| Secret detected | `continue=true` | Empty by default | Metadata event only | Original passes through |
| Classifier/shaper/verifier error or deadline | `continue=true` | Redacted reason code | Metadata event only | Original passes through |
| Ledger initialization/lock/write failure | `continue=true` | Redacted reason code | None | Original passes through; transformed result discarded |
| Response serialization failure | No partial JSON; exit 0 | Fixed reason code | None | Host keeps original |

The engine preallocates the receipt ID and serializes the complete Hook response
before persistence. The receipt transaction then commits before the prebuilt
`updatedToolOutput` response is written. SQLite uses a bounded busy timeout and
concurrency tests; no unbounded retry or sleep is allowed.

New receipts carry `delivery_state=prepared`. After Hook stdout writes and
flushes successfully, the adapter updates the receipt to `emitted`; write
failure leaves it prepared. `tgl stats` excludes prepared rows from its primary
aggregate and reports them separately. Even emitted savings are labeled
"governed candidate savings": stdout delivery cannot prove that a host accepted
the response or that a provider billed fewer tokens. Actual provider/model-input
savings require provider usage or the controlled live benchmark.

### 2. Evidence-backed decisions and risks

Each engine result must carry:

- `strategy`
- `confidence`
- `risk`
- `protected_fact_count`
- `preservation_check`
- `token_before`, `token_after`, and `tokens_saved`
- `receipt_id` only when the original was persisted

Risk rules:

- `low`: recognized content type, high-confidence parser, preservation checks
  pass, secret detection negative.
- `medium`: manual-only strategy or incomplete structural evidence; automatic
  path must pass through.
- `high`: secret/security/user-instruction match or preservation failure;
  automatic path must pass through and must not persist raw content.
- `unavailable`: classification could not be completed; automatic path passes
  through and reports no fake safety conclusion.

Only transformed receipts are described as auditable transformations. A
separate `governance_events` table retains privacy-safe operational evidence
using closed enums and numeric metadata only: timestamp, source kind, action,
risk, reason code, estimated token count, and optional receipt ID. It stores no
raw payload, payload hash, tool input, command, path, matched secret value, or
free-form message. `list_context_risks` reads this metadata surface and must not
imply that passthrough payload content is recoverable or inspected.

### 3. Privacy and ledger lifecycle

- Default persistence mode is `transformed_only`.
- Passthrough outputs create no receipt and store no original payload.
- Secret-like payloads create no receipt and store no original payload.
- The ledger remains local plaintext; README and `tgl doctor` must state this.
- `retention_days` becomes active through explicit `tgl prune`.
- `tgl stats` reports stored transformations only and labels token counts as
  estimated unless provider usage data is supplied.
- Existing v0.1 ledgers remain readable. Pruning may remove old rows only after
  the configured retention policy is active.

Ledger schema v2 uses `PRAGMA user_version=2` and a single transaction to:

- preserve the v0.1 `receipts` table and add nullable strategy/evidence fields
  required for new receipts
- create the payload-free `governance_events` table
- create indexes for created time, receipt ID, and retention pruning

Existing rows are labeled legacy and remain retrievable. They are not deleted
automatically because migration must not destroy user evidence. `tgl doctor`
reports the count of legacy rows and warns that v0.1 may contain passthrough or
secret-like plaintext. `tgl purge-legacy --confirm` is the explicit destructive
path. SQLite deletion/pruning is documented as logical deletion, not secure
erasure.

Retention semantics:

- `retention_days` is `null` or an integer from 1 through 3650; `null` disables
  pruning and zero/negative values are invalid.
- Cutoff uses UTC receipt creation time.
- `tgl prune` is explicit in v0.2; no startup deletion or concurrent implicit
  pruning occurs in the Hook path.
- Pruning receipts also removes linked metadata events in one transaction.
- Unlinked governance events use the same UTC cutoff and are pruned
  independently. `tgl prune` reports deleted receipt and event counts.

Ledger path resolution is singular: explicit CLI `--db`, then config
`ledger.path`, then the default `.tgl/ledger.sqlite` relative to the project
config directory. Relative backend and ledger paths resolve from the config
file directory, never the caller's incidental cwd.

Secret-like detection must include conservative patterns for common API keys,
authorization headers, private-key blocks, passwords, and known environment
variable names. The detector must never echo matched values into receipts,
logs, errors, or test output.

SQLite database, journal, and WAL files are all part of the same local plaintext
security boundary. Creation uses the narrowest practical current-user
permissions per platform. Any process running as the same user, and any MCP
client granted access to the restore server, may read governed originals. This
is an accepted and prominently documented v0.2 boundary, not an encryption or
same-user isolation guarantee.

MCP `retrieve_original` is paginated by default with `offset` and bounded
`max_chars`, returning truncation/remaining metadata. CLI retrieval may export
the exact full original outside model context. Retrieval verifies the stored
hash and returns a structured integrity error on mismatch.

### 4. Runtime configuration and installation

Add a typed config parser shared by CLI, Hook, MCP server, and Gateway.

Config is strict: unknown top-level or policy fields fail validation. Precedence
is CLI override, explicit config, then typed defaults. Runtime adapters receive
the parsed typed config rather than rereading individual fields.

Persistent Claude installation is supported only after
`npm install -g token-governance-layer`. The installer resolves and records
stable absolute Hook/MCP executable paths. When invoked through npm exec/npx, it
rejects `claude-install` with guidance to install globally; npx remains supported
only for non-persistent one-shot CLI commands such as help, manual govern, and
inspection against an explicit ledger.

Commands:

- `tgl init --project .`: create `token-governance.config.json` if absent.
- `tgl claude-install --project .`: idempotently install the hook and restore
  MCP server, creating config when absent and never overwriting user values.
- `tgl claude-uninstall --project .`: remove only TGL-owned hook/server entries;
  preserve config and ledger unless an explicit purge option is separately
  confirmed.
- `tgl doctor --project . --integration`: validate config, commands, Python
  version, hook smoke, standalone MCP initialize/list, and write permissions.
- `tgl claude-install --project . --repair`: explicitly repair incomplete
  TGL-owned installation state after read-only diagnosis.
- `tgl prune --config ...`: enforce retention and report deleted receipt count.

The safe install order is config, MCP restore server, Hook, then a payload-free
`.tgl/install-state.json` marked `complete`. Hook transforms only when install
state is complete; incomplete state always passes through. Synchronous failures
restore in-memory originals. A crash before Hook leaves restore-only state; a
crash after Hook but before `complete` still fails open. `doctor` is strictly
read-only and reports incomplete TGL-owned state. Repair requires explicit
`claude-install --repair`; removal requires explicit `claude-uninstall`.
Uninstall removes the Hook first, then MCP, then install state. Ownership is
recorded by exact normalized entries
and a TGL-managed command argument, not broad substring matching. Tests inject
failure at every write/replace boundary. Cross-file atomicity is not claimed.

### 5. MCP protocol hardening

Standalone server:

- Never respond to JSON-RPC notifications. Dispatch supported notifications
  (`initialized`, cancellation) for state changes and explicitly ignore
  unsupported notifications without response.
- Negotiate a supported protocol version rather than returning an unrelated
  hard-coded value.
- Preserve structured MCP tool errors without leaking payloads.

Experimental Gateway:

- Ignore client notifications correctly.
- Initialize every backend before `tools/list` or `tools/call`.
- Send `notifications/initialized` after successful backend initialization.
- Add bounded request timeouts and deterministic shutdown.
- Drain backend stderr without mixing it into stdio protocol output.
- Route responses by request ID and tolerate backend notifications.
- Fail one backend independently where possible and expose a structured
  unavailable reason.
- Document that v0.2 Gateway proxies tools only, not resources or prompts.

Gateway lifecycle contract:

```text
Client: CREATED -> INITIALIZED -> ACTIVE -> CLOSING -> CLOSED
Backend: STARTING -> INITIALIZED -> ACTIVE -> FAILED | CLOSING -> CLOSED
```

Requests before client initialize return protocol errors. A backend receives the
negotiated initialize request and `notifications/initialized` before any tool
request. The Gateway acts as its own MCP client with its own `clientInfo` and
declared capabilities; it never forwards or impersonates upstream client
identity. It validates each backend's selected protocol version and stores the
backend capabilities independently. A backend becomes ACTIVE only after a valid
initialize response and successful initialized notification; any failure closes
that backend process through the deterministic shutdown path. The upstream
client becomes ACTIVE only after its own initialized notification. Supported
protocol versions are centralized and checked against the official MCP contract
before implementation. Backend responses are routed by
request ID through a reader loop; backend notifications never satisfy pending
requests. Cancellation is forwarded when a backend request ID exists.
`notifications/tools/list_changed` invalidates that backend's cached catalog.
Backend stderr is drained into a secret-redacted 64 KiB ring buffer. Request
timeout defaults to 10 seconds and is configurable only from 1 through 120
seconds. Shutdown closes stdin, waits 2 seconds, terminates, waits 2 seconds,
then kills. Partial backend failure returns a structured unavailable reason and
does not report fake health for that backend.

JSON-RPC transport/protocol failures are returned as JSON-RPC errors. A backend
`tools/call` result, including MCP `isError`, is forwarded as a tool result and
must not be rewritten into a transport error.

Gateway stays outside the primary quick start and remains marked experimental
until strict-fixture tests pass. The core v0.2 release may proceed if real
backend interoperability remains incomplete, but Gateway is then omitted from
the feature list and carries an explicit experimental limitation. The
reproducible interoperability matrix uses pinned, credential-free reference
servers selected and recorded in `docs/gateway.md`; results are release
evidence, not an unqualified compatibility claim.

### 5.1 Public CLI and MCP contract

- `tgl govern` remains, but accepts only `--strategy auto|repetitive_log|
  test_output|build_output`; arbitrary `--content-type` generic summarization is
  removed.
- MCP `govern_context` remains with the same registered strategy enum and the
  same engine path. It cannot bypass secret detection, preservation, size, or
  persistence rules.
- The generic head/tail summarizer is deleted, not renamed or retained as a
  hidden manual mode.
- Gateway schema receipts continue to measure lazy schema exposure and are not
  described as content compression.

These are intentional v0.2 breaking MCP/CLI contracts. Old `content_type`
arguments and assumptions that `retrieve_original` returns unbounded plain text
receive fixed validation errors with migration guidance. Schemas, README, and
CHANGELOG must show the new strategy enum and paginated restore response.

### 6. Verification and benchmark evidence

Add deterministic safety fixtures for:

- secrets in the first, middle, and final output sections
- critical facts in middle lines
- repeated logs with unique errors
- pytest/unittest failure blocks
- npm/build output with warnings and failed targets
- Unicode, long lines, empty output, malformed hook payloads, and large output

Add an A/B benchmark harness that records raw and governed outcomes. Required
metrics:

- estimated and, when available, provider-reported input tokens
- task/test success
- number of agent turns and tool calls
- elapsed time
- restore count
- false-compression incidents

The release README may publish deterministic size-reduction data, but it must
label it a microbenchmark. Claims such as "same task quality" require a live
controlled benchmark with explicit spend approval and reproducible task data.

The deterministic benchmark command is
`python -m benchmarks.run --manifest benchmarks/fixtures/manifest.json --output benchmarks/results/v0.2.0.json`.
The manifest is versioned, the seed is fixed, and the result conforms to
`benchmarks/schema/result-v1.schema.json` with environment metadata and per-case
preservation evidence. CI runs
`python -m benchmarks.check_readme --results benchmarks/results/v0.2.0.json --readme README.md --readme README.zh-CN.md`
to reject README result drift.
Without paid live A/B approval, the README states that the microbenchmark proves
fixture size reduction and preservation checks only; it does not prove provider
cost reduction or unchanged task quality.

### 8. Release flow

1. Implement on `codex/v0.2-trustworthy-release`.
2. Run focused tests continuously and the full matrix before release.
3. Build and install the npm tarball in a clean temporary prefix.
4. Run `tgl init`, install, doctor, hook smoke, MCP smoke, uninstall, and
   reinstall from the packed artifact.
5. Review git diff and privacy scan tracked files plus package contents.
6. Run release preflight with TLS verification enabled: remove
   `NODE_TLS_REJECT_UNAUTHORIZED`, pass `npm ping`, authenticate as the confirmed
   dedicated publisher, verify npm owner/2FA or token scope, and verify GitHub
   push/tag/Release permissions. No tag is created before these checks pass.
7. Commit only v0.2 files and push the feature branch.
8. Fast-forward/merge to `main` only after all gates pass.
9. Tag `v0.2.0`, push the tag, and create a draft GitHub Release from the exact
   tag commit.
10. Publish the previously hashed tarball to npm, verify registry metadata, then
    publish the GitHub Release.
11. Verify GitHub commit, tag, npm version, npm package metadata, clean install,
    and final repository status.

A structured release script reads a `VERSION` source, updates `package.json`,
`pyproject.toml`, and runtime version constants, then verifies equality. Before
publish it records clean HEAD SHA, tag target, packed manifest, tarball SHA-256,
version, and privacy scan result. After publish it checks npm `dist-tags`,
`gitHead`, tarball integrity, and GitHub tag/Release SHA. If npm fails, the
GitHub Release remains an unpublished draft titled `INCOMPLETE v0.2.0` and the
public tag is not moved or reused. The controller may retry only the byte-exact
same tarball with the recorded SHA-256 after fixing transient authentication or
registry state. Any artifact-content change requires a new version and tag.

## Alternatives Considered

### Keep generic head/tail summarization and improve marker coverage

Rejected. More keyword markers cannot prove preservation of arbitrary source
or business facts. It would continue to optimize the numeric compression rate
at the cost of task correctness.

### Rewrite around the official MCP SDK in v0.2

Deferred. The npm package currently embeds dependency-free Python source; adding
the Python SDK would break the npm distribution model unless PyPI/uv packaging
is redesigned. v0.2 hardens the bounded tools-only implementation and keeps the
Gateway experimental. A packaging redesign can be evaluated after the core
Claude workflow is trustworthy.

### Add encrypted ledger storage now

Deferred. Encryption introduces key management and recovery semantics that do
not belong in this corrective release. The safer v0.2 default is not to persist
secret-like or passthrough payloads and to enforce retention.

## Migration or Compatibility Policy

- Existing SQLite ledgers remain readable.
- Existing TGL hook and MCP entries are replaced idempotently by
  `claude-install`.
- Existing user hooks and unrelated MCP servers are preserved.
- The v0.1 generic automatic compression behavior is intentionally not
  preserved. Unknown and source-oriented output passes through in v0.2.
- Package and command names remain unchanged.
- Gateway configuration remains accepted where practical, but Gateway is
  documented as experimental and no broad compatibility claim is carried.

## Acceptance Checks

- Official Hook schema evidence and the real Claude viability probe pass before
  implementation proceeds beyond the integration adapter.
- `python -m pytest -q` passes with new safety, config, retention, installer,
  MCP lifecycle, and integration tests.
- CI passes on Windows, macOS, and Linux for the declared Python/Node matrix.
- Secret-like probe: original output unchanged, no receipt, no database match.
- Secret fixtures assert that the receipt insert path is never called and scan
  database, journal, WAL, events, Hook stdout/stderr, and redacted backend stderr
  buffers for the matched value.
- Mixed stdout/stderr/exit-status fixtures preserve all required blocks.
- Middle-fact probe: unknown/file context unchanged, no receipt.
- Recognized repetitive-log and failing-test fixtures reduce size while exact
  protected facts remain byte-for-byte present.
- No automatically transformed payload receives `risk=low` without a passing
  preservation check.
- `retention_days` deletes expired rows and preserves non-expired rows.
- A real v0.1 database fixture migrates transactionally, retains legacy
  receipts, reports legacy risk, supports explicit prune/purge, and rolls back
  on injected migration failure.
- Gateway notification probe emits no response.
- Strict backend fixture rejects `tools/list` before initialize; Gateway test
  proves correct lifecycle.
- Backend stderr flood and timeout fixtures do not deadlock the Gateway.

## Risks

- `blocking`: Current npm publisher metadata exposes a personal email. The npm
  release must stop until a dedicated public identity/email is configured or
  the user explicitly accepts that exposure.
- `blocking`: Automatic transformation must not ship until secret, middle-fact,
  and preservation tests pass.
- `blocking`: No "same task quality" marketing claim without a controlled live
  benchmark and explicit spend approval.
- `blocking`: The real Claude Hook viability probe must prove model-visible
  replacement; otherwise product architecture returns for approval.
- `design_risk`: Secret detectors can miss novel formats. Mitigation is
  transformed-only storage, conservative content eligibility, documented local
  plaintext storage, and user-configurable additional patterns.
- `design_risk`: Hand-written MCP transport may remain less portable than an
  official SDK. Mitigation is experimental status, strict lifecycle tests,
  bounded timeouts, and real-backend interoperability evidence.
- `design_risk`: Narrower defaults may reduce reported savings. This is accepted
  because task correctness and trust outrank headline compression rate.
- `recorded_debt`: Model-specific tokenizers, encrypted ledger, Codex/Cursor
  adapters, resources/prompts proxying, dashboard, and cloud/team features.

## Open Questions

- Is a small paid live-agent A/B benchmark authorized? Without that approval,
  v0.2.0 will publish only deterministic microbenchmark claims and will not say
  that task quality is unchanged.

Resolved decisions:

- Failure of the real-backend Gateway matrix does not block the core release;
  Gateway remains experimental and is omitted from primary feature claims.
- Existing v0.1 ledger rows are retained with a warning; deletion is explicit.
- CLI/MCP manual governance remains only for registered safe strategies.
- Same-user local access to plaintext governed originals is an accepted and
